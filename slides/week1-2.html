Week 1-2: Why Traditional Data Management Broke & What Replaced It

From RDBMS → NoSQL → data lakes → lakehouse paradigm

Lecture Overview

This week explores the evolution of data management systems, the challenges of traditional approaches, and the emergence of modern paradigms. We'll revisit the 5 Vs of Big Data and draw from the 2024 paper "What Goes Around Comes Around... And Around..." by Michael Stonebraker and Andrew Pavlo (Carnegie Mellon University), focusing on RDBMS adaptations, NoSQL convergence, and lakehouses.

  • Understand why RDBMS struggled with scale and variety.
  • Learn about NoSQL's trade-offs and data lakes' flexibility.
  • Discover how lakehouses combine the best of lakes and warehouses.

Reference: Stonebraker & Pavlo (2024)

Evolution Timeline

A visual and historical overview of data management progression, integrating RDBMS genealogy, adaptations, and modern shifts from the paper and DBMS bloodlines:

RDBMS Era (1970s-2000s)

Structured data with SQL and ACID, but limited scalability for big data (Stonebraker & Pavlo, pp. 21-22). Dominance of relational model absorbing alternatives.

NoSQL Rise (2000s)

Addressed RDBMS scalability; features like JSON, BASE consistency (pp. 23-24). Convergence as NoSQL adds SQL-like interfaces.

Data Lakes (2010s)

Cheap raw storage in cloud object stores, but lacked governance (pp. 27-28).

Lakehouse Paradigm (2020s+)

Unifies lakes with ACID, open formats (e.g., Delta Lake, Iceberg); cloud-driven elasticity (p. 28).

RDBMS Genealogy

Tracing bloodlines from foundational systems (based on Hasso-Plattner-Institut RDBMS Genealogy):

  • 1970: E.F. Codd's relational model; IBM System R (first SQL).
  • 1974-1977: Ingres (UC Berkeley) → PostgreSQL (1996 → v17 2024), Redshift (2012), CockroachDB (2015), YugabyteDB (2017).
  • 1979: Oracle V2 → v23ai (2024).
  • 1981-1983: IBM SQL/DS and DB2 → v13 (2023).
  • 1986: Postgres (Berkeley) branches.
  • 1987: SQL Server (Sybase-based) → v2022 (preview v2025).
  • 1995: MySQL → v8.4 (2024).
  • 2000: SQLite → v3.46 (2024); Firebird → v5 (2023).
  • 2009: MariaDB → v11.0 (2024).
  • 2010: SAP HANA → v2.0 SPS07 (2023).
  • 2012: Amazon Redshift; Snowflake.
  • 2022: Google AlloyDB.
  • 2023: SQL:2023 with graph queries; AI integrations.

Family Tree Branches: System R (IBM, Oracle, SQL Server, Sybase, Teradata); Ingres (PostgreSQL, CockroachDB, etc.).

SQL Standards Evolution

Timeline of SQL standards, rooted in Codd's 1970 model (integrated with paper's note on SQL's persistence and extensions, pp. 21-23):

1970-1979

Codd's relational model; SEQUEL at IBM; first commercial: IBM System/38, Oracle V2.

1986: SQL-86

First ANSI standard.

1992: SQL-92

Major revision with entry-level features widely implemented.

1999: SQL:1999

Regular expressions, recursive queries, triggers, object-oriented features.

2016: SQL:2016

Row pattern matching, JSON operations (aligns with NoSQL convergence).

2023: SQL:2023

JSON data type, Property Graph Queries (SQL/PGQ).

Historical Context: Evolved from SQUARE to SEQUEL; standardized by ANSI/ISO. As of 2025, no new standards beyond SQL:2023.

NoSQL Evolution

Timeline and categories of NoSQL, emphasizing rise for scalability and convergence with RDBMS (from paper, pp. 23-25):

1960s-1990s

Early non-relational (hierarchical IMS, network CODASYL); "NoSQL" coined in 1998.

2004-2009

Google BigTable (2004, inspired HBase/Cassandra); CouchDB (2005, document); Hadoop/S3 (2006); Amazon Dynamo (2007, key-value like Riak); Cassandra (2008, wide-column); Neo4j (2008, graph); MongoDB/Redis (2009).

2020s

Hybrid/multi-model systems; AI integrations; convergence with SQL (e.g., MongoDB adds SQL in 2021).

Categories: Key-value (Redis), Document (MongoDB, CouchDB), Wide-column (Cassandra), Graph (Neo4j). Rise: Handled unstructured data, scale; trends: Edge computing, serverless, SQL/NoSQL blur.

The 5 Vs of Big Data Revisited

Challenges driving evolution (tied to paper's Big Data context, pp. 27-28):

Volume

Massive scales beyond RDBMS (addressed by lakes/lakehouses).

Velocity

Real-time needs (NoSQL strength).

Variety

Unstructured data (NoSQL/lakes).

Veracity

Quality/trust (lakehouse governance).

Value

Efficient insights (cloud elasticity).

Lessons Learned

From Stonebraker & Pavlo (pp. 21-32):

  • Innovation cycles: Alternatives converge to SQL/RM; good ideas absorbed (e.g., JSON in SQL).
  • Hardware-driven progress: Columnar storage, cloud, accelerators.
  • Niches persist: Graph/vector as extensions in RDBMS.
  • Cloud/open standards: Shared-disk revival, interoperability.
  • Hype caution: Marketing succeeds but engineering debt accumulates; migration risks limit NewSQL.
  • Future: Evolve with ML/AI, self-driving features.

systemctl status

Hands-on: Lab 0 - Personal Linux Data Workstation Setup

Set up your environment:

  1. Install Ubuntu or a Linux distro.
  2. Update packages: sudo apt update && sudo apt upgrade
  3. Install tools: sudo apt install tmux jq docker.io
  4. Enable Docker: sudo systemctl start docker
  5. Test: Run a container and process sample data.
