Lecture Overview

This week explores the evolution of data management systems, the challenges of traditional approaches, and the emergence of modern paradigms. We'll revisit the 5 Vs of Big Data and draw from the 2024 paper "What Goes Around Comes Around... And Around..." by Michael Stonebraker and Andrew Pavlo (Carnegie Mellon University), focusing on RDBMS adaptations, NoSQL convergence, and lakehouses.

  • Understand why RDBMS struggled with scale and variety.
  • Learn about NoSQL's trade-offs and data lakes' flexibility.
  • Discover how lakehouses combine the best of lakes and warehouses.

Reference: Stonebraker & Pavlo (2024)

Evolution Timeline

A visual overview of data management progression, integrating RDBMS genealogy, adaptations, and modern shifts from the paper and DBMS bloodlines:

1970s-2000s

RDBMS Era

Structured data with SQL and ACID, but limited scalability for big data (Stonebraker & Pavlo, pp. 21-22). Dominance of relational model absorbing alternatives.

2000s

NoSQL Rise

Addressed RDBMS scalability; features like JSON, BASE consistency (pp. 23-24). Convergence as NoSQL adds SQL-like interfaces.

2010s

Data Lakes

Cheap raw storage in cloud object stores, but lacked governance (pp. 27-28).

2020s+

Lakehouse Paradigm

Unifies lakes with ACID, open formats (e.g., Delta Lake, Iceberg); cloud-driven elasticity (p. 28).

Why RDBMS Struggled with Scale and Variety

Traditional RDBMS faced limitations in scaling for big data due to inability to distribute workloads across machines natively, leading to the rise of NoSQL (p. 29). They struggled with "one shot" tasks like text processing and ETL, requiring schema loading upfront (p. 22). Additionally, handling arrays and diverse data types was inefficient despite extensions like SQL/MDA (p. 25). Vertical scaling was insufficient due to cost and complexity. RDBMS also struggled with unstructured data variety and high velocity. They excel at consistency but struggle with high write throughput in OLTP contexts.

NoSQL Trade-offs and Data Lakes Flexibility

NoSQL prioritized scalability and flexibility over relational integrity, offering weaker consistency (BASE) and no initial ACID, leading to duplicated data and denormalization issues (p. 23). By late 2010s, most added SQL and ACID, converging with RDBMS. NoSQL handles unstructured/semi-structured data flexibly but trades consistency for scalability. Data lakes allow flexible storage without schema enforcement, using open formats for schema-on-read (p. 28). They support dynamic workloads and cloud-native deployments but introduce governance challenges.

How Lakehouses Combine Lakes and Warehouses

Lakehouses unify lakes' flexibility with warehouses' governance/performance, supporting SQL/non-SQL on object stores with open formats like Parquet/ORC (p. 28). They add ACID via middleware (Delta Lake, Iceberg, Hudi) for transactions/versioning (p. 28). Scalability via cloud elasticity and disaggregated storage; governance with metadata/caching (p. 28). Combines warehouses' reliability/performance with lakes' scalability/cost. Enables BI/ML on all data with open management. Unifies low-cost open storage with structured querying. Hybrid architecture for expansive storage with structure.

The 5 Vs of Big Data Revisited

Challenges driving evolution (tied to paper's Big Data context, pp. 27-28):

Volume

Massive scales beyond RDBMS (addressed by lakes/lakehouses).

Velocity

Real-time needs (NoSQL strength).

Variety

Unstructured data (NoSQL/lakes).

Veracity

Quality/trust (lakehouse governance).

Value

Efficient insights (cloud elasticity).

Lessons Learned

From Stonebraker & Pavlo (pp. 21-32):

  • Innovation cycles: Alternatives converge to SQL/RM; good ideas absorbed (e.g., JSON in SQL).
  • Hardware-driven progress: Columnar storage, cloud, accelerators.
  • Niches persist: Graph/vector as extensions in RDBMS.
  • Cloud/open standards: Shared-disk revival, interoperability.
  • Hype caution: Marketing succeeds but engineering debt accumulates; migration risks limit NewSQL.
  • Future: Evolve with ML/AI, self-driving features.

1.6 Purpose of Database Systems

Main Text: In the early days, database applications were built directly on top of file systems, which leads to:  Data redundancy and inconsistency: data is stored in multiple file formats resulting induplication of information in different files  Difficulty in accessing data • Need to write a new program to carry out each new task  Data isolation • Multiple files and formats  Integrity problems • Integrity constraints (e.g., account balance > 0) become “ buried ” in program code rather than being stated explicitly • Hard to add new constraints or change existing ones

1.7 Purpose of Database Systems (Cont.)

Main Text:  Atomicity of updates • Failures may leave database in an inconsistent state with partial updates carried out • Example: Transfer of funds from one account to another should either complete or not happen at all  Concurrent access by multiple users • Concurrent access needed for performance • Uncontrolled concurrent accesses can lead to inconsistencies  Ex: Two people reading a balance (say 100) and updating it by withdrawing money (say 50 each) at the same time  Security problems • Hard to provide user access to some, but not all, data Database systems offer solutions to all the above problems
